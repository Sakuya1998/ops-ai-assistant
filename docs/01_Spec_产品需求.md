## 2. 产品背景与业务目标

### 2.1 现状痛点
- 告警量大且噪声高，值班人员难以快速识别真实故障
- 故障排查跨指标、日志、事件，人工关联效率低
- 根因定位和修复依赖少数专家，知识沉淀不足
- 夜间高压值班下误操作风险高，恢复链路缺少标准化

### 2.2 产品愿景
构建基于 Eino 的运维AI助手，让系统在“发现问题—定位根因—执行修复—复盘沉淀”全链路具备智能协作能力。

### 2.3 业务目标（首发6个月）
- MTTR 降低 ≥ 60%
- 告警自动分诊率 ≥ 70%
- 常见故障自动修复成功率 ≥ 50%
- 高危操作误执行率 = 0
- 一线运维新人独立处置率提升 ≥ 40%

## 3. 用户角色与使用场景

| 角色 | 典型职责 | 核心诉求 |
|------|----------|----------|
| 值班工程师（On-call） | 接警、定位、升级 | 5分钟内得到可信根因方向 |
| SRE工程师 | 制定修复策略、保障稳定性 | 降低重复劳动，标准化修复流程 |
| 运维主管 | 组织协同、质量复盘 | 可审计、可衡量、可持续改进 |
| 新人运维 | 执行Runbook、学习处置流程 | 快速获得上下文与可执行步骤 |

## 4. 典型端到端流程

### 4.1 P0流程：告警驱动自动分诊
1. 告警通过 Webhook 进入助手
2. 助手聚合重复告警并识别影响范围
3. 自动拉取指标、日志与变更事件
4. 输出 Top N 根因假设与证据链
5. 给出“自动修复 / 人工确认 / 升级专家”建议

### 4.2 P1流程：对话驱动智能诊断
1. 用户以自然语言提问系统异常
2. 助手抽取实体（服务、时间窗、指标类型）
3. 触发工具查询并行分析
4. 输出结构化结论、影响评估、下一步动作

### 4.3 P1流程：修复与回滚闭环
1. 用户选择建议动作或手工下发操作
2. 助手执行权限校验和风险门禁
3. 执行前快照，执行后自动验证
4. 失败时按预设策略触发回滚
5. 自动生成复盘草稿并归档知识库

## 5. 用户故事与验收标准

### US-1：自然语言查询系统状态（P0）
**作为** 运维工程师  
**我希望** 通过自然语言快速获得异常解释  
**从而** 避免在多系统间来回切换排查

**验收标准（AC）**
- AC-1：支持中文/英文输入，并能识别中英混输服务名
- AC-2：可识别服务名、环境、时间范围、指标类型
- AC-3：纯对话推理响应时间 P95 < 3 秒（不含外部查询耗时）
- AC-4：输出结构化结论（状态级别、影响面、建议动作）
- AC-5：结论需附证据摘要与可信度分数

### US-2：智能告警分析与根因定位（P0）
**作为** 值班工程师  
**我希望** 告警触发后自动完成关联分析  
**从而** 缩短从接警到定位的时间

**验收标准（AC）**
- AC-6：支持 Prometheus/Alertmanager 告警 Webhook 接入
- AC-7：默认关联最近 30 分钟核心指标与日志
- AC-8：输出 Top 3 根因假设，并按置信度排序
- AC-9：每条假设必须包含至少一条可追溯证据来源
- AC-10：对重复告警进行聚合去重并输出主告警ID

### US-3：自动化故障修复（P1）
**作为** SRE工程师  
**我希望** 常见故障可以半自动/自动修复  
**从而** 减少重复人工操作和误操作

**验收标准（AC）**
- AC-11：支持配置化 Playbook 与参数模板
- AC-12：高风险操作默认强制人工确认
- AC-13：执行前完成关键状态快照
- AC-14：执行后自动进行健康检查与指标回归验证
- AC-15：失败场景支持一键回滚并记录原因

### US-4：运维知识库问答（P1）
**作为** 新入职运维人员  
**我希望** 复用历史故障处置经验  
**从而** 快速学习并独立处理常见问题

**验收标准（AC）**
- AC-16：支持 Incident、Runbook、架构文档统一检索
- AC-17：回答必须展示引用来源（文档链接/工单ID）
- AC-18：支持多轮追问与上下文保持
- AC-19：无法给出答案时返回升级路径与推荐专家

### US-5：多智能体协作处理复杂故障（P2）
**作为** 运维主管  
**我希望** 跨领域复杂故障可并行协作诊断  
**从而** 提高复杂问题处置效率与稳定性

**验收标准（AC）**
- AC-20：支持网络/数据库/应用/基础设施专业Agent
- AC-21：主控Agent自动拆解任务并并行分派
- AC-22：聚合报告可显示各Agent分结论和冲突点
- AC-23：支持人工接管或指定专家Agent优先执行

## 6. 功能需求分层（MVP优先级）

| 模块 | 功能项 | 优先级 | 首发要求 |
|------|--------|--------|----------|
| 智能分诊 | 告警接入、去重聚合、影响分析 | P0 | 必须 |
| 智能诊断 | 指标日志关联、根因假设、证据输出 | P0 | 必须 |
| 交互体验 | 对话式问答、结构化卡片结果 | P0 | 必须 |
| 执行动作 | 修复建议、审批门禁、执行回执 | P1 | 建议 |
| 知识系统 | RAG问答、来源追溯、复盘沉淀 | P1 | 建议 |
| 多Agent协作 | 专家Agent并行、冲突融合 | P2 | 可延后 |

### 6.1 In Scope（本期范围）
- 告警接入与智能分诊
- 指标+日志联合诊断
- 可控自动化修复（含审批、回滚）
- 运维知识库问答与来源追踪
- 操作审计与最小权限控制

### 6.2 Out of Scope（本期不做）
- 替代现有监控系统本身（Grafana/Prometheus）
- 自动容量规划与成本优化
- 攻击溯源与安全威胁处置平台
- 全量跨云统一资源编排

## 7. 非功能需求（NFR）与SLO

| 维度 | 指标要求 | SLO | 备注 |
|------|----------|-----|------|
| 可用性 | 服务可用性 | 月度 ≥ 99.9% | 含降级可用 |
| 交互性能 | 对话响应时延 | P95 < 5秒 | 不含慢查询外部依赖 |
| 分诊时效 | 告警初判产出 | P95 < 30秒 | 从Webhook接收开始计时 |
| 诊断质量 | 根因Top3命中率 | ≥ 75% | 以复盘标注为准 |
| 安全合规 | 高危操作误执行 | 0次 | 强制审批+审计 |
| 可观测性 | 关键链路可追踪率 | 100% | Trace + 审计日志 |

## 8. 安全、权限与审计要求

- 所有执行类工具必须进行 RBAC 权限校验
- 高风险动作必须满足“二次确认 + 审批人身份记录”
- 任何自动修复都必须具备“前置快照 + 后置验证 + 可回滚”
- 审计日志必须覆盖“谁在何时对什么执行了什么操作，结果如何”
- 输出给用户的内容必须脱敏，不得泄露密钥、令牌、凭据

## 9. 交互与输出规范

### 9.1 诊断结果标准结构
- 结论等级：正常 / 警告 / 严重
- 影响范围：服务、租户、区域、时间窗
- 根因假设：至少1条，最多3条，含置信度
- 证据清单：指标/日志/事件引用
- 动作建议：立即动作、观察动作、升级动作

### 9.2 修复动作展示规范
- 标注风险等级：低/中/高
- 标注预估影响：服务中断、性能抖动、无感变更
- 显示前置条件：权限、依赖、维护窗口
- 展示执行结果：成功/失败、回滚状态、验证摘要

## 10. 发布策略与验收门槛

### 10.1 版本发布策略
- Alpha：仅内部SRE小范围试用，验证链路可用性
- Beta：生产影子流量验证，开启人工审批保护
- GA：覆盖核心业务线，启用SLO考核与周复盘机制

### 10.2 GA准入门槛
- P0功能验收通过率 = 100%
- 连续2周无高危误执行
- 真实告警样本下，MTTR 改善达到目标下限（≥ 40%）
- 审计链路完整率 = 100%
